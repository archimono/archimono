# Contributing to archimono

Thanks for your interest in contributing! This guide covers the essentials.
The full developer contract (coding style, docstring policy, citation
requirements) lives in [CLAUDE.md](CLAUDE.md).

## Development setup

```bash
git clone https://github.com/archimono/archimono
cd archimono
pip install -e ".[dev]"      # add ,docs to also build the documentation
```

## Running the tests

Tests are organised into three tiers:

```bash
pytest               # fast + contract (default; excludes heavy)
pytest -m fast       # inner dev loop (< 10s)
pytest -m contract   # pre-commit gate (< 3 min)
pytest -m heavy      # full robustness (CI or manual, < 10 min)
pytest -m ""         # all tiers
```

## Code style

- Follow the [Google Python Style Guide](https://google.github.io/styleguide/pyguide.html).
- Every public module, class, and function gets a Google-style docstring
  with `Args:`, `Returns:`, and `Raises:` sections where applicable.
- All function signatures must be fully annotated; `mypy --strict` must pass.
- Any algorithm or mathematical result taken from a paper, textbook, or
  external source must be cited (author, title, year, and the specific
  equation or section) in the docstring or an inline comment.

Before opening a pull request, make sure these pass:

```bash
ruff check src/      # lint (E, F, W, I, UP, ANN at line length 88)
mypy src/            # type-check (strict)
pytest               # tests
```

## Submitting changes

1. Create a feature branch off `main`.
2. Keep changes surgical and focused; add or update tests for any behaviour
   change.
3. Ensure lint, type-check, and tests pass.
4. Open a pull request describing what changed and why.

## Documentation

The published documentation is an autogenerated API reference built from
source docstrings with Sphinx. Build it locally with:

```bash
pip install -e ".[docs]"
sphinx-build -b html docs docs/_build/html
```
